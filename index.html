Case study · 01

The best ad teams spend $50M a year on creative and still can't keep up with the volume their media buyers need.

Project · Creative IQ · Agentic AI · AdTech
The story

Every modern performance marketing team runs into the same wall. Their media buyers can run 40 ad variations a day. Their creative team can produce 4.

This gap isn't a staffing problem. It's an architecture problem. Today's creative stack is humans-with-tools: designers in Figma, editors in Premiere, a stray Midjourney license here and there. Generative video models are now good enough to close the gap, but nobody's wired them up the way a media buyer actually thinks about creative — as a portfolio to be optimized, not individual files to be commissioned.

Creative IQ is the orchestration layer. Brief goes in, performance-tuned variations come out, every asset tagged against the brand book and the KPI it's being optimized for. Instead of "make me a video," it's "make me 30 variations of this hook, optimized for three demographics and two platforms, none of which violate the brand guardrails."

How it's built

An async pipeline of specialist agents.

The core insight is that creative production isn't one model's job. It's a graph of jobs: concept generation, script writing, shot planning, video synthesis, voiceover, brand-compliance audit, performance scoring. Each node is an agent with one job it does well.

Brief & brand book→Concept agent
↓ N variations per brief, parallelized
Concept agent→Script→Shot plan→Video synthesis
↓ Sora / Runway / Kling, async queue
Rendered assets→Brand-compliance ML→PASS / REWRITE
↓ Flagged assets loop back to the concept agent with the reason
Approved assets→Perf scoring→Feedback into next brief

The non-obvious part is the feedback loop. Most creative tools treat each asset as a one-shot artifact. Creative IQ treats the performance of last week's assets as context for next week's concept generation. The system gets smarter about what works for each brand, automatically.

OrchestrationMulti-agent pipeline on EKS
Video modelsSora / Runway / Kling
InfraTerraform IaC, GitHub Actions
ComplianceCustom brand-guideline classifier
QueueAsync job system for long renders
ObservabilityPer-agent latency, cost, quality tracking
Where it is today

Active development. The agent graph ships variations end-to-end, compliance ML is catching most brand violations in testing, and the infra is production-grade (Terraform-managed EKS, CI/CD via GitHub Actions).

What's next
  • Per-brand fine-tuning. The compliance classifier generalizes poorly. Moving to per-brand LoRA adapters so each client's rules are learned, not prompted.
  • Platform-native rendering. Reels, TikTok, and Shorts each have different aspect, timing, and audio conventions. Right now the pipeline handles these with prompts; it should handle them with dedicated post-processing nodes.
  • Cost-aware routing. Sora is gorgeous and expensive. Kling is fast and cheaper. A routing layer that picks the right model per shot based on budget + creative importance.

Case study · 02

Clinic staff spend 45 minutes Ctrl+F'ing through PDFs to answer a single prior auth question.

Project · AuthLookup · GenAI · Healthcare
The story

Prior authorization is a $35B/year administrative burden, and almost all of it is the same five questions asked a million times: does this patient's insurance require pre-auth for this procedure, what documentation do they need, and which diagnoses qualify?

The answers live in 90MB of CMS policy PDFs. The people who need them are nurses and billing staff on a phone with a patient. The gap between those two facts is where the $35B goes. Cohere Health raised $200M solving a slice of this; Olive AI raised even more and didn't. The problem isn't going away.

AuthLookup is the plain-English interface to CMS. Ask "brain MRI with contrast, Medicare" and get back the CPT code, whether prior auth is required, the documents you need, the diagnoses that qualify, the common denial reasons, and a FHIR R4 CRD/DTR response ready to plug straight into the EHR.

How it's built

A RAG pipeline on real CMS data, running entirely local.

The hard part wasn't the model. It was the data. No public API exists for payer policies, so I pulled CMS bulk downloads directly: Local Coverage Determinations, Billing Articles, and coverage CSVs. Parsed 4,179 CPT codes with their actual Medicare coverage requirements. Built body-part-aware matching so "knee MRI" resolves to lower-extremity codes, not brain. Added contrast-protocol logic so 70551 and 70553 don't get confused.

CMS bulk data (90MB)→Parser & cleaner→Structured policy DB
↓ 4,179 CPT codes, body-part tagged, contrast-aware
User query (plain English)→Ollama (local LLM)→Intent + CPT candidates
CPT candidates→ChromaDB retrieval→Relevant policy chunks
Policy + query→Ollama synthesis→Answer + FHIR R4 response
↓ Everything runs on-device. Zero API calls. PHI never leaves the machine.

The architectural call that matters most: running it 100% locally on Ollama. Healthcare shops can't send PHI to OpenAI. They can't afford enterprise API costs at query volumes that look like 100K/day. Local inference solves both at once, which is why AuthLookup is deployable on-prem today, not "after we figure out HIPAA."

What's next
  • Commercial payers. Medicare is the easy win because the data is public. UHC, Aetna, and Cigna policy docs are harder to get but where the volume lives.
  • EHR integration depth. FHIR R4 responses are generated, but plugging into Epic's or Cerner's prior-auth flow end-to-end is a separate engineering effort.
  • Graph over vector. Payer policies reference each other constantly. A GraphRAG upgrade on Neo4j would catch the cross-references vector search misses, especially for bundled procedures.

Case study · 03

A third of the food we grow rots before anyone eats it — and most of it rots because people are stuck coordinating logistics on phone calls.

Project · FreshFlow · Agentic AI · CPG
The story

Food distribution is the rare industry that's both enormous and analog. A mid-sized distributor coordinates across hundreds of farms, a fleet of trucks, and dozens of retailers, and most of the coordination still happens on the phone.

When a farm reports unexpected surplus strawberries, a human has to call retailers until one says yes. When a truck breaks down, a human has to re-route by hand. When demand spikes because of a weather forecast, a human has to remember which supplier has capacity. Every one of these is a coordination loop that an agent can run better, faster, and continuously.

FreshFlow is a multi-agent platform that runs those loops. Agents handle demand forecasting, spoilage risk scoring, and rerouting negotiations, turning a manual phone-tree operation into a continuously re-planning system. Built with a team of five in 72 hours for the Amazon Nova Hackathon.

How it's built

Three specialist agents, one coordination loop.

The instinct with multi-agent is to build one god-agent that does everything. That's wrong. In logistics, the agents need to match the real roles: a forecaster who predicts demand, a risk scorer who tracks spoilage windows, a negotiator who reroutes inventory. Each one has clear inputs, clear outputs, and clear KPIs. They coordinate through a shared state store, not through chat.

Supplier feeds→Forecast agent→Predicted demand by SKU/region
Inventory + timestamps→Spoilage risk agent→Risk scores per pallet
↓ Shared state: demand + risk + current routes
Shared state→Reroute agent→Reroute proposals + expected savings
Proposals→Human approval (Streamlit)→Execute

The interesting design choice was keeping a human in the loop for execution. In logistics, wrong decisions compound fast (one bad reroute cascades through a week's worth of deliveries), so we made the system propose rather than execute. The dashboard shows the proposed action, the expected savings, and the reasoning trace. Humans click approve. Trust gets built on the right side of the risk curve.

Agent frameworkAmazon Nova multi-agent
ForecastingProphet + gradient-boosted residuals
Risk scoringFeature-engineered classifier
OptimizationLLM + constraint solver hybrid
UIStreamlit with live proposal feed
Team5 engineers, 72 hours
What's next
  • Real supplier data. The hackathon version runs on synthetic distributor data. The next version needs real logistics integrations — SAP, NetSuite, or the smaller distributor-specific TMS systems.
  • Learned negotiator. The reroute agent uses rules + an LLM today. With real data, a learned model would negotiate better trade-offs between cost, risk, and relationship value.
  • Observability for multi-agent. When three agents disagree on a state, today you have to read logs. That's not going to scale. Needs a purpose-built trace viewer for agent coordination.

Case study · 04

Every PM has the same ritual: scroll through 200 support tickets, 40 Slack threads, and 12 call transcripts, and try to remember what users actually want.

Project · Cursor for PMs · GenAI · Product Intelligence
The story

Product managers are drowning in unread user feedback. The feedback exists. The patterns exist. The problem is that the human brain can't hold 300 support tickets in working memory while writing a PRD.

The current tools don't help. Support platforms let you read tickets one at a time. Analytics dashboards show counts, not causes. Notion AI summarizes a single document. None of them do the thing PMs actually need: remember everything users have said, connect it to what they're saying now, and cite the evidence when you claim a pattern.

Cursor for PMs is that memory layer. Ingest every feedback channel — support tickets, reviews, Slack threads, call transcripts — into a persistent context graph. Ask "what are users really saying about feature X?" and get back an answer with cited evidence. Then turn that insight into a spec, and that spec into agent-executable code.

How it's built

An agentic retrieval graph, not a chatbot over docs.

V1 was a plain RAG pipeline. It was wrong. The reason is that PM questions aren't lookup questions. They're reasoning questions. "Are complaints about onboarding up or down this quarter vs. last?" requires aggregation, not retrieval. "What are the three biggest themes in churn interviews?" requires clustering, not keyword match.

V2 rebuilt the retrieval layer as a LangGraph of specialist steps: topic modeling, temporal aggregation, evidence chaining, and only then synthesis. The LLM plans which steps to invoke based on the question type. The output always includes citations back to the original feedback. If it's not cited, it's not claimed.

Support + Slack + reviews + calls→Ingestion workers→Normalized feedback store
↓ pgvector + pg full-text + topic clusters + time index
PM question→Query planner (LangGraph)
↓ Decides: lookup? aggregation? clustering? temporal compare?
Plan→Retrieval→Aggregation→Clustering
Results + raw quotes→Synthesis with citations→Answer

The architecturally important decision was storing feedback in three representations at ingestion: vector embedding (for semantic search), full-text index (for exact phrase match), and topic cluster assignment (for "what are the themes"). RAG-over-PDFs teaches you to reach for vector similarity as the hammer. PM questions taught me most of them aren't nails.

What's next
  • From insight to spec to code. Today the system answers questions. The next version generates PRD drafts and, eventually, coding-agent-executable specs.
  • Fine-tuned embeddings per customer. Product-specific jargon ("churn," "activation," internal feature names) embeds poorly on general models. Per-customer embedding fine-tunes would sharpen retrieval.
  • Proactive alerts. Right now the PM asks. The system should notice: "Onboarding complaints spiked 3x this week; the pattern matches the release of feature Y."
